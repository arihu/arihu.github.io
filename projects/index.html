IMDb Runtime Scraper
(Currently outdated as IMDb changed the layout of their website)

A webpage that will take a given IMDb TV Show id and return the total runtime in minutes as given in each episode's runtime description with BeautifulSoup and Selenium.

Almost all online sites that calculate the runtime of a show uses the formula # of episodes * Average runtime of all episodes. This can lead to inaccurate results especially for shows with varying runtimes for each episode(more common because of streaming).

The scraper works by first receiving the IMDb id, retrieves the HTML of the episode list containing that id with BeautifulSoap as the episode list is loaded statically. Next, the scraper uses Selenium to load each episode page dynamically, scrape the runtime, and sum the results.

Imager API

Build an API that allows someone to register and authenticate with a username and password with Spring Security, and store images(encoded in base64) into a MySQL database using Hibernate. Also used Postman to check my API. The first step in interacting with this API is to register with a username, email and password.

Once the user is registered, the AuthenticationController would pass the Request and return the JSON Web Token generated by the JWT Service. Next, The AuthenticationService would store the user info to the User Repository. Hibernate will then put the info into the MySQL Database.

The next step for this API is to authenticate their identity by sending a post request with their username and password. This would generate a following JWT token that can be used for posting or getting images.

After authenticating, one can send a post request to post an image. You can authenticate by pasting the token to "bearer token" or "Basic Auth" and typing the username and password.

After posting the image, it will appear in the image table in the MySQL database.

You can get all images, by authenticating like the post request. It will return JSON of the image and user ID, title, image in base64 format, and description.

Other API endpoints include GET image, which when given the image id (ex: image2/3), will return that specific image. Put image, which will replace the image with the given image ID, and DEL image which will delete the image with the given image ID.

Itube

Followed a tutorial to create a youtube clone, ITube, where users can sign in with their google accounts using Firebase Auth, upload videos when authenticated, and watch videos that are uploaded. Google cloud is used to run the Docker files of the API and Video Processing, and Firebase is used as the database. The frontend incorporates React, Typescript, and NextJS, which would send requests to a backend that uses ExpressJS. I deviated from the tutorial by using TailwindCSS for the frontend, Typescript for the API service, and implementing a upload form. ITube's Upload Page features a form for Title, Description, Upload Video, and Upload Thumbnail. The current iteration only supports Video uploads.

Once the video is uploaded, it will call an endpoint in the public video api service, which is received by Cloud Storage by a implementing public api service which incorporates Firebase functions. Once Cloud Storage receives the video, it triggers the Pub/Sub subscription to queue a message to process the video with the video processing service.

The video processing service is a Docker image created to process the video to 360 or 720p, store the result in a processed bucket, and store video data to Firebase.

For the videos to be displayed to the front end, the web client first gets videos from the public api service, which will retrieve the data from firestore. The videos are then mapped to the front page.

Clicking on the video brings ITube to the watch page, where users can watch the video.

Glass Database

Planned with a group of 4 to create a database for a hypothetical grocery store chain, Glass. Glass is a small local grocery store chain with low customer and supplier count, 5 employees(1 manager, 2 associates, 2 janitors), a customer population at 200, inventory of 210 products, and 150 purchases a day. In Phase 1 of our project, we created a Proposal detailing the assumptions made for our database and the Entity Relationship Dictionary and Diagram based on those assumptions.

In Phase 2, we made corrections to the Entity Relationship Dictionary and diagram, and applied the Relational Mapping Algorithm to convert the diagram into Schema Diagram and Dictionary.

In Phase 3, we started applying our database schema into MySQL. We created an implementation log to implement our schema using CREATE TABLE. We then use the DESCRIBE command in MySQL Shell to check if all the tables matched our assumptions.

In the final phase, we populate our database according to our assumptions (ex: 3 Stores with 3 different addressess, 200 customers, 150 different orders, 210 different products with unique attributes...), and provided query examples to access specific attributes of our database.

Twenty Card Solver

Build a Lisp/Prolog function/clause that finds the longest play with least amount of Kings or largest point value in a hand of a card game Twenty, where given a hand, one must match an incrementing count that reaches twenty using a subset of that hand. Kings are also counted as wildcards, face cards can have pairs, and cards have different point values but Kings have a point penalty. This was done by breaking the problem into chunks. To get all possible plays, Copy all cards that are smaller than the count, generate pairs if applicable, generate wild cards up to count, and combine. To find the longest play with the least amount of Kings, I sort all possible plays by a score that prioritizes length with a larger value than Kings. To find the largest point value, I sort all possible plays by point value. The image below is both behaviors playing each other in Lisp using DrRacket.
